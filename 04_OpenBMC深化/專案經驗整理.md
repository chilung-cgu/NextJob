> 面試時最重要的就是能講出你做過的事！這份指南幫你包裝經驗

---

## 📌 STAR 法則

```
面試回答專案問題時，使用 STAR 法則：

S - Situation（情境）：背景是什麼？
T - Task（任務）：你負責什麼？
A - Action（行動）：你做了什麼具體的事？
R - Result（結果）：結果如何？學到什麼？
```

## 🔷 常見專案經驗範例（供你參考）

### 範例 1：新增 Sensor 支援

```
Situation：
我們的伺服器需要支援新的電源供應器（PSU），
客戶要求能透過 Redfish 監控 PSU 的溫度和功率。

Task：
我負責在 OpenBMC 中新增這款 PSU 的感測器支援，
包括 I2C 驅動設定、hwmon 整合，以及 D-Bus 配置。

Action：
1. 閱讀 PSU 的 datasheet，了解 I2C address 和暫存器定義
2. 在 device tree 中加入 PSU 的 I2C device 節點
3. 設定 kernel 使用對應的 pmbus driver
4. 新增 phosphor-hwmon 設定檔，定義 sensor label
5. 驗證 D-Bus 上可以正確讀取數值
6. 和 Redfish 團隊確認 API 回傳正確

困難：
一開始 sensor 數值跳動很大，經過調查發現是 SMBus timeout，
調整了 I2C clock frequency 後解決。

Result：
- 成功上線，客戶滿意
- 學會了 pmbus driver 的運作機制
- 建立了新增 sensor 的標準流程文件
```

### 範例 2：Debug I2C 問題

```
Situation：
生產線回報某批板子的 BMC 無法讀取 DIMM 溫度，
但其他 sensor 正常。

Task：
負責分析和解決這個問題。

Action：
1. 先用 i2cdetect 確認 I2C bus 狀態
2. 發現 SPD (DIMM 的 EEPROM) 位址有回應，但讀取 timeout
3. 拿示波器量 I2C 波形，發現 clock stretching 過長
4. 查閱 DIMM datasheet，確認是 DIMM 回應太慢
5. 調整 BMC 的 I2C timeout 設定
6. 和硬體團隊確認這批 DIMM 規格不同

Result：
- 問題解決，生產線恢復正常
- 更新了 I2C driver 的 timeout 預設值
- 撰寫了 I2C troubleshooting 文件
- 學會使用示波器分析 I2C 問題
```

### 範例 3：效能優化

```
Situation：
客戶反映 BMC 啟動時間太長，約 90 秒，
要求縮短到 60 秒以內。

Task：
分析開機流程，找出可優化的環節。

Action：
1. 使用 systemd-analyze 分析啟動時間
2. 發現 phosphor-xxx 服務順序有問題
3. 調整 systemd service 的 After/Requires
4. 將非必要服務改為延遲啟動
5. 優化 udev rules 減少設備掃描時間

Result：
- 開機時間從 90 秒降到 55 秒
- 學會了 systemd 的各種調優技巧
- 建立了開機效能測量的 benchmark
```

## 📝 準備問題清單

面試官可能問的深入問題：

**技術細節：**
```
□ 這個功能用了哪些技術/API？
□ 為什麼選擇這個方案而不是其他？
□ 遇到什麼困難？怎麼解決的？
□ 如果重做一次，會有什麼改進？
□ 這個功能的性能/品質如何保證？
```

**合作和溝通：**
```
□ 你和誰一起做這個專案？
□ 如何分工？
□ 有沒有和其他團隊溝通？
□ 遇到意見不同怎麼處理？
```

**學習和成長：**
```
□ 從這個專案學到什麼？
□ 有什麼失敗的經驗嗎？
□ 如果再做一次會怎麼做？
```

## 💡 包裝技巧

```
1. 量化成果：「縮短 30%」比「變快」有說服力
2. 技術關鍵字：適度使用專業術語展示深度
3. 問題解決：強調你如何分析和解決問題
4. 團隊合作：展示你能和別人協作
5. 持續學習：說明你從中學到什麼
6. 不要說謊：面試官會追問細節
```
